If you're building AI agents in TypeScript, you've probably come across [Mastra](https://mastra.ai/docs). It's one of the most active agent frameworks in the ecosystem, with a clean API for defining [agents](https://mastra.ai/docs/agents/overview), [tools](https://mastra.ai/docs/agents/using-tools), and multi-agent workflows. Teams at Replit and Fireworks use it in their stacks.

The thing is, a framework gives you building blocks, not a place to run them. You still need deployment, observability, state management, and infrastructure around your agents before they're useful to anyone.

That's the gap [Agentuity](/blog/agentuity-v1-is-here) fills. It's a full-stack platform for AI agents: [deployment](/ai-agent-deployment), [observability](/product/observability), cloud services, and an [AI Gateway](/product/ai-gateway). It complements the framework you're already using. Bring your Mastra agents, wrap them in a few lines, and deploy with one command. And if you don't need a framework at all, you can build agents [directly on the Agentuity SDK](https://agentuity.dev/) too.

## The Integration Pattern

The idea is straightforward: define your agent with Mastra, then wrap it with Agentuity's `createAgent()`. Mastra handles the agent logic and LLM calls. Agentuity handles the rest: deployment, persistent state, routing, and [infrastructure](/ai-agent-infrastructure).

Here's what that looks like. This is a conversational agent with memory, where Mastra runs the conversation and Agentuity's [thread state](/product/storage) persists history across requests:

```typescript
import { createAgent } from '@agentuity/runtime';
import { s } from '@agentuity/schema';
import { Agent } from '@mastra/core/agent';

const chatAgent = new Agent({
  id: 'chat-agent',
  name: 'Chat Agent',
  instructions: 'You are a helpful assistant. You remember previous conversations within this thread.',
  model: 'openai/gpt-4o-mini',
});

export default createAgent('chat', {
  schema: {
    input: s.object({ message: s.string() }),
    output: s.object({ response: s.string() }),
  },
  handler: async (ctx, { message }) => {
    const history = (await ctx.thread.state.get('messages')) ?? [];
    const result = await chatAgent.generate([...history, { role: 'user', content: message }]);

    await ctx.thread.state.push('messages', { role: 'user', content: message }, 20);
    await ctx.thread.state.push('messages', { role: 'assistant', content: result.text }, 20);

    return { response: result.text };
  },
});
```

Mastra's `Agent` class gives you the LLM layer. Agentuity's `createAgent()` gives you everything around it: typed schemas, thread-isolated state, structured logging via `ctx.logger`, and automatic [observability](/product/observability). When you're ready, run `agentuity deploy` and you're live.

## Zero-Config LLM Access

One thing worth calling out: when you deploy a Mastra agent on Agentuity, you don't need to bring your own OpenAI API key. The [AI Gateway](/product/ai-gateway) provides unified LLM access, and a small bridge file redirects Mastra's OpenAI-compatible client to it automatically:

```typescript
// src/lib/gateway.ts
if (!process.env.OPENAI_API_KEY && process.env.AGENTUITY_SDK_KEY) {
  const gw = process.env.AGENTUITY_AIGATEWAY_URL ?? 'https://agentuity.ai';
  process.env.OPENAI_API_KEY = process.env.AGENTUITY_SDK_KEY;
  process.env.OPENAI_BASE_URL = `${gw}/gateway/openai`;
}
```

Import this once and every Mastra agent in the project routes through Agentuity's gateway. One place for model access, usage tracking, and billing.

## What You Get from the Platform

Mastra gives you the framework for building agents. Agentuity gives you the platform for running them. In practice, that means:

- **Observability** — Every invocation is traced automatically. Logs, latency, token usage, and session history show up in the [dashboard](/product/observability) with no instrumentation code.
- **Thread State** — Persistent, thread-isolated [storage](/product/storage) via `ctx.thread.state`. This is what makes the memory pattern above work, and it extends to approvals, preferences, and any state your agent carries between requests.
- **Cloud Services** — [KV, vector search, and object storage](/product/storage) accessible from `ctx`. No provisioning, no connection strings.
- **AI Gateway** — Unified LLM access through a [single gateway](/product/ai-gateway). Switch models, track usage, and manage billing in one place.
- **Agent-to-Agent Networking** — Deploy multiple Mastra agents and have them [call each other](/blog/agent-to-agent-networking) through Agentuity's router.

### Tool Calling with Real APIs

The simplest way to extend a Mastra agent is with tools. Mastra's `createTool()` gives you typed inputs, descriptions for the LLM, and an `execute` function. The agent decides when to call the tool based on the user's message.

Here's a weather tool that hits the Open-Meteo API — no API key required:

```typescript
import { createTool } from '@mastra/core/tools';
import { Agent } from '@mastra/core/agent';
import { z } from 'zod';

const weatherTool = createTool({
  id: 'get-weather',
  description: 'Fetches current weather for a location',
  inputSchema: z.object({
    location: z.string().describe('The city or location to get weather for'),
  }),
  execute: async ({ location }) => {
    const coords = await getCoordinates(location);
    if (!coords) return `Could not find location: ${location}`;

    const data = await fetch(
      `https://api.open-meteo.com/v1/forecast?latitude=${coords.lat}&longitude=${coords.lon}&current=temperature_2m,weather_code,wind_speed_10m`
    ).then(r => r.json());

    return `${coords.name}: ${data.current.temperature_2m}°C, Wind: ${data.current.wind_speed_10m} km/h`;
  },
});

const weatherAgent = new Agent({
  id: 'weather-agent',
  instructions: 'You are a helpful weather assistant. Use the get-weather tool to fetch weather data.',
  model: 'openai/gpt-4o-mini',
  tools: { weatherTool },
});
```

The Agentuity wrapper stays thin — delegate to the Mastra agent, return the result:

```typescript
export default createAgent('weather', {
  schema: {
    input: s.object({ message: s.string() }),
    output: s.object({ response: s.string(), tokens: s.number() }),
  },
  handler: async (ctx, { message }) => {
    const result = await weatherAgent.generate(message);
    return { response: result.text, tokens: result.usage?.totalTokens ?? 0 };
  },
});
```

Mastra handles the function calling loop. Agentuity handles deployment and schema validation. The `using-tools` example also includes an activities agent that combines weather data with a second tool to suggest things to do — showing how multiple tools compose naturally.

### Structured Output with Zod

When you need the LLM to return data in a specific shape — not just freeform text — Mastra's `structuredOutput` option paired with a Zod schema handles it cleanly. The `structured-output` example builds a day planner that returns typed time blocks and activities:

```typescript
import { z } from 'zod';

const DayPlanSchema = z.object({
  plan: z.array(
    z.object({
      name: z.string().describe('Time block name (e.g., Morning, Afternoon)'),
      activities: z.array(
        z.object({
          name: z.string(),
          startTime: z.string().describe('HH:MM format'),
          endTime: z.string().describe('HH:MM format'),
          description: z.string(),
          priority: z.enum(['high', 'medium', 'low']),
        })
      ),
    })
  ),
  summary: z.string(),
});

const result = await plannerAgent.generate(
  `Plan type: work. Schedule my day: I have two meetings and need to write a design doc.`,
  { structuredOutput: { schema: DayPlanSchema } }
);

const plan = result.object; // Fully typed — plan.plan[0].activities[0].priority is 'high' | 'medium' | 'low'
```

The interesting part here is the dual-schema approach. Zod schemas drive Mastra's structured output from the LLM. Agentuity's `s` schemas validate the API input/output layer. They mirror each other, but serve different purposes — one is for the model, the other is for the platform.

The Agentuity handler stores plan history in thread state, so users can see past plans and track token usage over time:

```typescript
await ctx.thread.state.push('history', {
  prompt: truncate(prompt, 50),
  timestamp: new Date().toISOString(),
  tokens,
  planType,
  activityCount: totalActivities,
}, 5);
```

### Input & Output Processors

Production agents need guardrails. Mastra's processor system lets you chain safety checks on both sides of the LLM call — before the message reaches the model and after the response comes back. The `processors-agent` example wires up a full pipeline:

```typescript
import {
  ModerationProcessor,
  PIIDetector,
  PromptInjectionDetector,
  TokenLimiterProcessor,
  UnicodeNormalizer,
} from '@mastra/core/processors';

const moderatedAgent = new Agent({
  id: 'moderated-agent',
  instructions: 'You are a helpful assistant.',
  model: 'openai/gpt-4o-mini',
  inputProcessors: [
    new UnicodeNormalizer({ stripControlChars: true, collapseWhitespace: true }),
    new PromptInjectionDetector({
      model: 'openai/gpt-4o-mini',
      threshold: 0.8,
      strategy: 'rewrite',
      detectionTypes: ['injection', 'jailbreak', 'system-override'],
    }),
    new PIIDetector({
      model: 'openai/gpt-4o-mini',
      strategy: 'redact',
      detectionTypes: ['email', 'phone', 'credit-card'],
    }),
    new ModerationProcessor({
      categories: ['hate', 'harassment', 'violence', 'self-harm'],
      threshold: 0.7,
      strategy: 'block',
    }),
  ],
  outputProcessors: [
    new TokenLimiterProcessor({ limit: 1000, strategy: 'truncate' }),
    new ModerationProcessor({
      categories: ['hate', 'harassment', 'violence'],
      threshold: 0.7,
      strategy: 'block',
    }),
  ],
});
```

Five processors in a chain. Input flows through unicode normalization (prevents homoglyph attacks), prompt injection detection (catches jailbreak patterns), PII redaction (strips emails, phone numbers, credit cards), and content moderation (blocks harmful categories). Output passes through token limiting and a second moderation check.

If any processor trips, Mastra sets a `tripwire` on the result and the Agentuity handler returns a structured error:

```typescript
const result = await moderatedAgent.generate(text);

if (result.tripwire) {
  ctx.logger.warn('Request blocked', {
    reason: result.tripwire.reason,
    processorId: result.tripwire.processorId,
  });
  return { response: result.tripwire.reason, success: false, /* ... */ };
}
```

The agent also tracks processing history in thread state — useful for auditing which inputs were flagged and why.

### Human-in-the-Loop Tool Approval

Some tools shouldn't execute without a human saying yes. Mastra supports this natively — you can flag specific tools as requiring approval, or enable approval for all tool calls at the agent level. The `agent-approval` example implements both patterns.

The flow works like this: when a Mastra agent wants to call a sensitive tool, it suspends instead of executing. The agent's `finishReason` comes back as `'suspended'`, and you get a `runId` that you can use to resume later:

```typescript
const result = await approvalAgent.generate(text, { requireToolApproval });

if (result.finishReason === 'suspended' && result.runId) {
  // Store the pending approval in thread state
  const pendingApproval = {
    toolName,
    toolCallId: suspendPayload.toolCallId,
    reason: 'This tool permanently deletes user data. Review before proceeding.',
    runId: result.runId,
    status: 'pending',
  };
  await ctx.thread.state.set('pendingApproval', pendingApproval);

  return { response: `Tool "${toolName}" requires approval.`, suspended: true, pendingApproval };
}
```

When the user approves or declines, Mastra resumes the agent from exactly where it left off:

```typescript
// Approve — tool executes and agent continues
const result = await approvalAgent.approveToolCallGenerate({ runId: pending.runId });

// Decline — agent acknowledges the tool was not executed
const result = await approvalAgent.declineToolCallGenerate({ runId: pending.runId });
```

The example includes four tools across a safety spectrum: `get-weather` and `search-records` execute immediately, while `delete-user-data` and `send-notification` always require approval. Agentuity's thread state stores the pending approval and the full approval history, so you get an audit trail for free.

### Multi-Agent Networks

For tasks that need coordination between multiple specialists, Mastra lets you compose agents into networks. The `network-agent` example sets up a routing agent that delegates to a research agent and a writing agent, with a weather tool available for simple lookups:

```typescript
import { Agent } from '@mastra/core/agent';
import { Memory } from '@mastra/memory';
import { LibSQLStore } from '@mastra/libsql';

const researchAgent = new Agent({
  id: 'research-agent',
  instructions: 'Gather concise research insights in bullet-point form.',
  model: 'openai/gpt-4o-mini',
});

const writingAgent = new Agent({
  id: 'writing-agent',
  instructions: 'Turn research into well-structured written content.',
  model: 'openai/gpt-4o-mini',
});

const routingAgent = new Agent({
  id: 'routing-agent',
  instructions: `You are a network of writers and researchers.
    For complex tasks, use the research agent to gather facts, then the writing agent to produce content.
    For simple weather questions, use the weather tool.`,
  model: 'openai/gpt-4o-mini',
  agents: { researchAgent, writingAgent },
  tools: { weatherTool },
  memory: new Memory({
    storage: new LibSQLStore({ id: 'network-store', url: 'file:mastra.db' }),
  }),
});
```

The routing agent uses LLM reasoning to decide which sub-agents to call, in what order, and how to combine their outputs. Ask it to "write a travel guide for Lisbon" and it'll route to the research agent for facts, then hand those to the writing agent to produce prose.

On the Agentuity side, the handler passes `ctx.thread.id` into Mastra's memory scope so conversation context persists across requests:

```typescript
const result = await routingAgent.generate(message, {
  memory: {
    resource: ctx.sessionId,
    thread: ctx.thread.id,
  },
});
```

### Network-Level Approval and Suspend/Resume

The `network-approval` example combines everything: multi-agent routing, tool approval, and suspend/resume into a single system. It's the most advanced pattern and demonstrates three distinct execution types within one network:

**Immediate execution** — safe tools like `search-web` and `lookup-info` run without interruption:

```typescript
// No approval needed — execute and return
const toolResult = await executeTool(toolName, toolArgs);
const followUpResult = await networkMastraAgent.generate([...conversationMessages, toolResult]);
```

**Approval-required** — sensitive tools like `delete-records` and `send-notification` pause for human review:

```typescript
if (TOOLS_REQUIRING_APPROVAL.has(toolName)) {
  await ctx.thread.state.set('pendingApproval', {
    toolName, toolCallId, toolArgs,
    subAgent: TOOL_SUB_AGENTS[toolName], // Maps tool → conceptual sub-agent
    conversationState: JSON.stringify(conversationMessages),
  });
  return { suspended: true, suspendType: 'approval' };
}
```

**Suspend/resume** — the `request-confirmation` tool pauses with a payload (options, context) and waits for user-provided data before continuing:

```typescript
if (TOOLS_WITH_SUSPEND.has(toolName)) {
  const suspendPayload = getSuspendPayload(toolName, toolArgs);
  // Payload includes: message, action description, and options for the user to choose from
  await ctx.thread.state.set('suspendedExecution', {
    toolName, suspendPayload: JSON.stringify(suspendPayload),
    conversationState: JSON.stringify(conversationMessages),
  });
  return { suspended: true, suspendType: 'suspend', suspendedExecution };
}
```

When the user responds, the resume function picks up the stored conversation state and continues:

```typescript
export async function resumeNetwork(suspended, resumeData, thread) {
  const conversationMessages = JSON.parse(suspended.conversationState);
  const toolResult = executeConfirmationWithResumeData(toolArgs, resumeData);

  const followUpResult = await networkMastraAgent.generate([
    ...conversationMessages,
    { role: 'tool', content: [{ type: 'tool-result', toolCallId, result: toolResult.data }] },
  ]);
  // Clean up state, store in history, return response
}
```

The key insight is that Agentuity's thread state acts as durable storage for the entire conversation flow. The conversation messages, pending approvals, and suspended executions all survive across HTTP requests, so a user can close their browser, come back later, and approve or resume exactly where they left off.
